# BizBoard — ベンチマーク参考情報

## 評価実施ガイドライン

### 評価環境の統一

各エージェントの実装を公平に比較するため、以下の環境を統一してください。

| 項目 | 指定 |
|------|------|
| Node.js | v20.x LTS |
| PostgreSQL | 15.x 以上 |
| OS | 評価者の環境に統一 |
| テスト時のブラウザ | Chrome 最新版 |

### 評価手順

#### Step 1: セットアップ確認

各エージェントの成果物に対して以下を実行:

```bash
# 1. 依存関係インストール
npm install

# 2. 環境変数設定
cp .env.example .env
# → .env を編集（DB接続情報等）

# 3. DBセットアップ
npx prisma migrate dev

# 4. シードデータ投入
npx prisma db seed

# 5. ビルドテスト
npm run build

# 6. 開発サーバー起動
npm run dev
```

各ステップでエラーが発生した場合、その内容を記録してください。

#### Step 2: 機能テスト

以下のテストケースを各エージェントの実装に対して実行:

**認証テスト**
- [ ] ログイン画面が表示される
- [ ] 正しい認証情報でログインできる
- [ ] 不正な認証情報でエラーが表示される
- [ ] ログアウトできる
- [ ] 未認証でのアクセスがリダイレクトされる

**ダッシュボード**
- [ ] KPIカードが表示される
- [ ] 数値がDBのデータと整合する

**営業管理**
- [ ] 顧客一覧が表示される
- [ ] 顧客を新規作成できる
- [ ] 顧客情報を編集できる
- [ ] 顧客を削除できる
- [ ] 商談一覧が表示される
- [ ] 商談のステータスを変更できる

**案件管理**
- [ ] 案件一覧が表示される
- [ ] 案件を新規作成できる
- [ ] タスクを追加・編集できる
- [ ] 進捗率が正しく表示される

**財務管理**
- [ ] 経費申請一覧が表示される
- [ ] 経費を申請できる
- [ ] Manager/Adminで承認/却下できる
- [ ] 承認ステータスが正しく反映される

**人事管理**
- [ ] 従業員一覧が表示される
- [ ] 従業員を登録・編集できる
- [ ] 部署管理ができる

#### Step 3: コードレビュー

以下の観点でコードを確認:

1. **型安全性**: `grep -r "any" src/` で `any` 使用箇所を確認
2. **制約違反**: `@ts-ignore`, `@ts-nocheck` の使用有無
3. **ディレクトリ構成**: ファイル一覧の確認
4. **コンポーネント設計**: Server/Client Componentの使い分け
5. **エラーハンドリング**: try-catch, error.tsx, loading.tsx の有無

## 参考: 一般的なAIエージェント評価の観点

### SWE-bench との違い

本評価はSWE-benchのような「既存バグの修正」ではなく、「ゼロからの実装」を評価します。

| 観点 | SWE-bench | 本評価 |
|------|-----------|--------|
| タスク | バグ修正 | フルスタック実装 |
| 入力 | Issue + 既存コード | 設計書 |
| 出力 | パッチ | 完全なプロジェクト |
| 評価 | テスト通過率 | 5軸の総合評価 |

### 注意事項

- 各エージェントへの指示は完全に同一のプロンプトを使用すること
- エージェントとの対話中に追加のヒントや修正指示を出さないこと（1回のプロンプトで完結）
- 実行環境（マシンスペック、ネットワーク等）の違いは動作性評価に影響しないよう考慮
- エージェントの応答時間や会話ターン数は今回の評価対象外（ただし参考情報として記録推奨）

### 参考記録項目（任意）

| 項目 | 説明 |
|------|------|
| 応答時間 | プロンプト送信から完了までの時間 |
| ターン数 | エージェントとの対話回数 |
| 生成ファイル数 | 出力されたファイルの総数 |
| 総行数 | 出力されたコードの総行数 |
| エラー修正回数 | 自己修正が必要だった回数（該当する場合） |
