# BizBoard — 評価基準

## 評価概要

各AIエージェントの実装結果を5軸・合計100点で評価します。

---

## A. タスク完了率（25点）

実装指示に対して、どの程度の機能が実装されているかを評価します。

### 採点基準

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| A-1. 認証機能 | 3点 | ログイン/ログアウト (1点)、ロールベースアクセス制御 (2点) |
| A-2. ダッシュボード | 3点 | KPIカード表示 (2点)、アクティビティフィード (1点) |
| A-3. 営業管理 | 5点 | 顧客CRUD (2点)、商談CRUD (2点)、売上グラフ (1点) |
| A-4. 案件管理 | 5点 | 案件CRUD (2点)、タスクCRUD (2点)、進捗表示 (1点) |
| A-5. 財務管理 | 5点 | 経費CRUD (2点)、承認ワークフロー (2点)、収支サマリー (1点) |
| A-6. 人事管理 | 4点 | 従業員CRUD (2点)、部署管理 (1点)、勤怠サマリー (1点) |

### 採点ガイド
- **完全実装**: 全配点を付与
- **部分実装**: 機能の骨格はあるが一部動作しない → 配点の半分
- **未実装**: 0点

---

## B. 操作品質（15点）

「表示される」ではなく「実際に操作してエラーが出ない」かを評価します。
（参照: `design/quality-guidelines.md` セクション1）

### 採点基準

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| B-1. CRUDフロー完走 | 5点 | 作成→一覧反映→詳細→編集→削除の一巡が各モジュールで途切れないか |
| B-2. エッジケース耐性 | 4点 | 空データ表示 (1点)、不正ID→404 (1点)、権限不足→適切な制御 (1点)、戻る/進む耐性 (1点) |
| B-3. エラーハンドリングUI | 3点 | error.tsx配置 (1点)、loading.tsx配置 (1点)、トースト/エラーメッセージ (1点) |
| B-4. フォームバリデーション | 3点 | 必須項目の空送信防止 (1点)、形式チェック (1点)、サーバーサイドバリデーション (1点) |

### テスト手順
1. 各モジュールでCRUD一巡を実施
2. データが0件の画面を確認
3. URLに存在しないIDを入力してアクセス
4. Memberロールで管理者操作を試行
5. フォームの必須項目を空で送信
6. ブラウザの戻るボタンでフォーム送信後の画面を確認

---

## C. セキュリティ（15点）

業務アプリとして最低限のセキュリティ対策がなされているかを評価します。
（参照: `design/quality-guidelines.md` セクション2）

### 採点基準

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| C-1. 認証チェック | 5点 | 全APIルート/Server Actionでセッション検証を実施しているか |
| C-2. 認可チェック | 4点 | ロールチェックがAPIレイヤーで行われているか（UI出し分けのみは0点） |
| C-3. 入力バリデーション | 3点 | サーバーサイドでバリデーション実装 (2点)、Zodなど型安全なバリデーション (1点) |
| C-4. 情報漏洩防止 | 3点 | パスワードハッシュ化 (1点)、エラー詳細非露出 (1点)、環境変数の適切な管理 (1点) |

### テスト手順
1. ブラウザのDevToolsからAPIを未認証で直接呼び出す → 401が返るか
2. Memberロールのトークンで管理者APIを呼び出す → 403が返るか
3. フォーム入力に `<script>alert(1)</script>` を入力 → そのままHTMLレンダリングされないか
4. エラー発生時のレスポンスにスタックトレースが含まれていないか
5. `.env.example` にシークレットの実値が含まれていないか

---

## D. パフォーマンス（10点）

明らかなパフォーマンス問題を作り込んでいないかを評価します。
（参照: `design/quality-guidelines.md` セクション3）

### 採点基準

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| D-1. N+1クエリ防止 | 3点 | リレーションデータの取得に `include`/`select` を使用しているか |
| D-2. ページネーション | 3点 | 一覧画面で `take`/`skip` による制限を行っているか |
| D-3. Server/Client分離 | 2点 | データ取得をServer Componentで行い、インタラクションのみClient Componentにしているか |
| D-4. バンドル・キャッシュ | 2点 | 不要な巨大importがないか (1点)、適切なrevalidation設定 (1点) |

### テスト手順
1. Prismaクエリログを有効化し、一覧画面でN+1が発生していないか確認
2. 一覧画面のAPIレスポンスが全件返却していないか確認
3. `"use client"` が付いたファイルでPrismaやDB直接アクセスをしていないか確認
4. `npm run build` の出力でバンドルサイズを確認

---

## E. 保守性・拡張性（10点）

チームで保守する業務アプリとして、将来の変更に耐えうる設計かを評価します。
（参照: `design/quality-guidelines.md` セクション4）

### 採点基準

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| E-1. 型定義の一元管理 | 3点 | Prisma生成型を活用 (2点)、不要な手書き型がない (1点) |
| E-2. 共通コンポーネント | 3点 | テーブル・フォーム等のUI部品が再利用可能な形で実装されているか |
| E-3. 定数・設定の管理 | 2点 | マジックナンバー排除 (1点)、ステータス値等の定数定義 (1点) |
| E-4. 命名・構成の一貫性 | 2点 | 命名規則の統一 (1点)、ディレクトリ構成の論理性 (1点) |

### チェック方法
1. `grep -r "any" src/` で `any` 使用箇所を確認
2. 同じ型定義が複数ファイルに存在しないか確認
3. 似た構造のコンポーネントがコピペされていないか確認
4. ステータス値がリテラル文字列でハードコードされていないか確認

---

## F. 自律的設計判断（15点）

基本設計から詳細設計への展開において、エージェントが行った設計判断の妥当性を評価します。

### 採点基準

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| F-1. 状態管理の選択 | 4点 | ライブラリの選定妥当性 (2点)、実装パターン (2点) |
| F-2. DB設計の展開 | 4点 | スキーマの正確性 (2点)、シードデータの質 (2点) |
| F-3. UIライブラリの選択 | 4点 | 選定の妥当性 (2点)、活用度 (2点) |
| F-4. アーキテクチャ判断 | 3点 | API設計 (1点)、データフロー (1点)、ディレクトリ構成 (1点) |

### 評価の観点
- プロジェクト規模に対して過不足ない技術選定をしているか
- 設計書にない詳細部分を合理的に補完しているか
- 選定した技術を適切に活用しているか（入れただけでなく）

---

## G. セットアップ・開発体験（10点）

開発者がプロジェクトを理解・セットアップする際の体験を評価します。

### 採点基準

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| G-1. ビルド・起動 | 4点 | `npm install` 成功 (1点)、マイグレーション成功 (1点)、シード成功 (1点)、`npm run build` 成功 (1点) |
| G-2. README | 3点 | セットアップ手順 (1点)、機能説明 (1点)、環境構築手順 (1点) |
| G-3. 環境・可読性 | 3点 | `.env.example` (1点)、適切なコメント (1点)、構造の分かりやすさ (1点) |

---
