# BizBoard — 品質ガイドライン

> 「とりあえず動く」ではなく「本番運用に耐える」品質を求める。本ガイドラインは必ず遵守すること。

---

## 1. 操作品質 — 全操作フローでエラーが起きないこと

| # | 要件 |
|---|------|
| OQ-01 | 全画面が正常に表示される（500エラー、白画面、ハイドレーションエラーなし） |
| OQ-02 | 全CRUDフローが完走する（作成→一覧反映→詳細→編集→削除） |
| OQ-03 | フォーム空送信でクラッシュせず、エラーメッセージが表示される |
| OQ-04 | 存在しないIDへのアクセスで404表示（500エラーではない） |
| OQ-05 | 認証フロー（ログイン→操作→ログアウト→再ログイン）が途切れない |
| OQ-06 | 権限不足時にクラッシュせず、適切なエラーまたはリダイレクト |
| OQ-07 | ページネーションが正しく動作する |
| OQ-08 | ブラウザの戻る/進むで壊れない |
| OQ-09 | データ0件で「データがありません」と表示され、エラーにならない |

**エラーハンドリング原則**: ユーザー操作の結果は必ず「成功」か「理解可能なエラーメッセージ」。未処理例外の露出は不可。
- APIルート: 全ルートで try-catch + 適切なHTTPステータスコード
- エラーバウンダリ: `error.tsx` をモジュール単位で配置
- ローディング: `loading.tsx` またはスケルトンUI

---

## 2. セキュリティ — 一般的な脆弱性がないこと

| # | 要件 |
|---|------|
| SQ-01 | 全API/Server Actionでセッション検証（未認証→401） |
| SQ-02 | ロールチェックをAPIレイヤーで実施（UI出し分けだけは不可） |
| SQ-03 | Prisma ORMのパラメータバインディング使用（`$queryRawUnsafe` 禁止） |
| SQ-04 | `dangerouslySetInnerHTML` 禁止 |
| SQ-05 | パスワードはbcrypt等でハッシュ化（平文保存厳禁） |
| SQ-06 | エラーレスポンスにスタックトレース・DB情報を含めない |
| SQ-07 | `NEXT_PUBLIC_` の適切な使い分け（シークレットをクライアントに露出しない） |
| SQ-08 | サーバーサイドで入力バリデーション必須（クライアントのみは不可） |

---

## 3. パフォーマンス — 明らかな問題を作り込まないこと

| # | 要件 |
|---|------|
| PQ-01 | N+1クエリ禁止（`include` / `select` を使用） |
| PQ-02 | 一覧画面は必ずページネーション（全件取得禁止） |
| PQ-03 | データ取得はServer Component、インタラクションのみClient Component |
| PQ-04 | 不要な再レンダリングを避ける（適切な粒度でのComponent分割） |
| PQ-05 | 巨大ライブラリの丸ごとimportを避ける |

---

## 4. 保守性・拡張性 — チームで保守する前提の設計

| # | 要件 |
|---|------|
| MQ-01 | Prisma生成型を活用（同じ型を複数箇所で手書きしない） |
| MQ-02 | ビジネスロジックとUI表示を分離 |
| MQ-03 | テーブル・フォーム等の共通UIコンポーネントを抽出 |
| MQ-04 | 命名規則をプロジェクト全体で統一 |
| MQ-05 | ステータス値・ページサイズ等を定数定義（マジックナンバー禁止） |
| MQ-06 | モジュール（営業/案件/財務/人事）ごとにディレクトリ整理 |
