# AIコーディングエージェント比較検証

複数のAIコーディングエージェントに同一の業務アプリケーション設計書を渡し、1回のプロンプトで実装を依頼。成果物の品質を7軸・100点満点で評価しました。

## 課題アプリ: BizBoard

営業・案件・財務・人事の4領域を統合管理する業務ダッシュボード。中規模フルスタックWebアプリケーション（想定20〜30ファイル）。

- **技術スタック**: Next.js App Router + Prisma + PostgreSQL + NextAuth（設計書で指定）
- **評価方法**: コードレビュー + 実際のビルド・起動・ブラウザ操作テスト
- **条件**: 全エージェントに同一のプロンプトを1回のみ渡し、追加指示なし

## 主要な発見

### 1. ブラウザで実際に動くエージェントはほぼなかった

7エージェント中、ブラウザからフォーム操作（ログイン・データ登録等）が正常にできたのは**Kiroの新規作成・削除・承認のみ**。他の6エージェントは全てフォーム入力が機能しません。

| 問題パターン | 該当エージェント |
|---|---|
| @base-ui/react Inputがrefを転送しない | Cursor, Claude Code |
| 自前InputにforwardRefがない | Codex |
| 編集ページでServer→Client関数渡し違反 | Kiro, Codex |
| ビルド自体が失敗 | GitHub Copilot, Windsurf |
| 実装範囲が不足（フォームUI未作成） | Antigravity |

コードレビューやビルドでは発見できず、**実際にブラウザで操作して初めて発覚する**問題です。全エージェントに共通して「自己検証（ブラウザでの動作確認）」が不足していました。

### 2. コード品質と動作品質は別物

Codexはコードレビューレベルでは最高品質（認証/認可一貫、Zodバリデーション、共通コンポーネント充実、any/ts-ignore使用ゼロ）でしたが、ブラウザで触ると基本操作が一切できませんでした。

### 3. モデル性能とスコアは必ずしも相関しない

最上位モデル（Claude Opus 4.6、GPT-5.4）を使用したエージェントが必ずしも上位ではなく、Kiro（Claude Sonnet 4.5）が首位。**モデル単体の性能よりも、エージェントのツール実装（ビルド確認、ライブラリ互換性チェック、ブラウザ操作検証）が結果を左右**しました。

### 4. 下位グループはビルドすら通らない

GitHub Copilot（exportエラー）とWindsurf（Prisma型エラー）はnpm run buildが失敗し、起動すらできませんでした。

## 結論

現時点のAIコーディングエージェントは、**コードの骨格を高速に生成する能力は高い**ものの、**ブラウザで実際に操作して動作検証する能力が不足**しています。特にUIライブラリの互換性問題やNext.jsのServer/Client境界制約など、静的解析やビルドでは検出しにくいランタイムエラーの検出が課題です。

実務での活用にあたっては、エージェントの出力をそのまま納品するのではなく、**人間によるブラウザ操作テストが必須**です。エージェント選定においてはモデル性能だけでなく、自己検証機能の有無を重視すべきです。
